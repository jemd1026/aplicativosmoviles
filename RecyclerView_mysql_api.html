PARTE 1: Arquitectura general

Como funciona esta arquitectura

A diferencia de SQLite que guarda datos dentro del dispositivo, o Firebase que provee un servicio gestionado en la nube, aqui se construye una API REST propia que expone los datos de una base MySQL a traves de HTTP. La app Android no toca MySQL directamente: siempre habla con la API.

Flujo completo

Android App
→
API REST (PHP)
→
MySQL

La API puede vivir en tu maquina local durante el desarrollo, o en un VPS/hosting cuando se despliega. Android solo necesita saber la URL base de la API.

SQLite

Local, dentro del dispositivo. Sin internet. Sin servidor.

Firebase Firestore

Nube gestionada por Google. Sin servidor propio. Cuotas incluidas.

MySQL + API REST

Tu propio servidor. Control total de los datos. Requiere backend propio.

Internet requerido No Si Si

Consideracion importante antes de empezar

Durante el desarrollo la API estara en tu maquina local (por ejemplo http://192.168.1.x/api/). Para que el emulador de Android Studio acceda a tu maquina local usa la IP 10.0.2.2 en lugar de localhost. Un celular fisico conectado al mismo WiFi usa la IP local de tu PC.

Cuando el proyecto se despliega en un VPS solo cambia la URL base en el archivo de configuracion de Android. Ese es precisamente el valor de usar un archivo .env en el servidor y una constante de URL configurable en el cliente.

PARTE 2: Preparar MySQL y la tabla usuario

Opciones de servidor

Para esta guia se puede usar un entorno local (XAMPP o WAMP en Windows, MAMP en Mac) o un VPS con acceso SSH. Ambas opciones usan exactamente el mismo codigo PHP y el mismo esquema de base de datos; solo cambia la URL que Android usara para conectarse.

Local con XAMPP Desarrollo

Instala XAMPP. Inicia Apache y MySQL desde el panel. PhpMyAdmin en localhost/phpmyadmin.

VPS propio Produccion

Acceso por SSH. MySQL instalado en el servidor. Dominio o IP publica para la URL de la API.

Hosting compartido

Panel cPanel o similar. MySQL y PHP disponibles. La URL de la API es el dominio del hosting.

Paso 2.1: Crear la base de datos

En PhpMyAdmin o desde la consola MySQL ejecuta:

CREATE DATABASE IF NOT EXISTS app_usuarios CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

Paso 2.2: Crear la tabla usuario

USE app_usuarios; CREATE TABLE IF NOT EXISTS usuario ( id INT NOT NULL AUTO_INCREMENT PRIMARY KEY, nombre VARCHAR(80) NOT NULL, apellido VARCHAR(80) NOT NULL, email VARCHAR(120) NOT NULL UNIQUE, password VARCHAR(255) NOT NULL, telefono VARCHAR(20) DEFAULT NULL, activo TINYINT(1) NOT NULL DEFAULT 1, created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

Sobre el campo password

La contrasena nunca se guarda en texto plano. En la API PHP se aplica password_hash() antes de insertar, y se verifica con password_verify(). El campo en MySQL es VARCHAR(255) porque el hash generado por PASSWORD_BCRYPT tiene 60 caracteres pero la longitud recomendada es 255 para soportar futuros algoritmos.

La columna password nunca se incluye en las respuestas GET de la API: el modelo PHP la excluye explicitamente al serializar.

Resultado esperado de esta parte

  • Base de datos app_usuarios creada en MySQL.
  • Tabla usuario con los campos indicados.
  • MySQL accesible desde PHP (local o VPS).

PARTE 3: API REST en PHP con archivo .env

Por que usar un archivo .env

Un archivo .env separa la configuracion del codigo. Las credenciales de la base de datos, el nombre de la BD y el modo de entorno se guardan ahi. El codigo PHP lo lee al arrancar. Cambiar de entorno local a VPS solo requiere editar .env, no el codigo fuente.

El proyecto de esta guia vive en D:\laragon\www\api_ucv\ y se accede desde Laragon en http://localhost/api_ucv/.

Paso 3.4: config/Database.php

Lee el .env sin dependencias externas y abre la conexion PDO. Si APP_DEBUG=true muestra el mensaje de error real; en produccion lo oculta.

<?php // config/Database.php class Database { private PDO $conn; public function __construct() { $this->loadEnv(__DIR__ . '/../.env'); $host = $_ENV['DB_HOST'] ?? '127.0.0.1'; $port = $_ENV['DB_PORT'] ?? '3306'; $name = $_ENV['DB_NAME'] ?? ''; $user = $_ENV['DB_USER'] ?? ''; $pass = $_ENV['DB_PASS'] ?? ''; $dsn = "mysql:host=$host;port=$port;dbname=$name;charset=utf8mb4"; try { $this->conn = new PDO($dsn, $user, $pass, [ PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION, PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC, PDO::ATTR_EMULATE_PREPARES => false, ]); } catch (PDOException $e) { $debug = ($_ENV['APP_DEBUG'] ?? 'false') === 'true'; http_response_code(500); echo json_encode([ 'error' => 'Error de conexion a la base de datos', 'detalle' => $debug ? $e->getMessage() : null, ]); exit; } } public function getConnection(): PDO { return $this->conn; } private function loadEnv(string $path): void { if (!file_exists($path)) return; $lines = file($path, FILE_IGNORE_NEW_LINES | FILE_SKIP_EMPTY_LINES); foreach ($lines as $line) { $line = trim($line); if ($line === '' || str_starts_with($line, '#')) continue; // Soporta valores con = dentro (ej. contrasenas con =) [$key, $value] = array_map('trim', explode('=', $line, 2)); // Quita comillas simples o dobles opcionales alrededor del valor $value = trim($value, '"\''); // Elimina comentarios inline (valor # comentario) if (($pos = strpos($value, ' #')) !== false) { $value = trim(substr($value, 0, $pos)); } $_ENV[$key] = $value; putenv("$key=$value"); } } }

Paso 3.5: models/Usuario.php

listar() y obtener() seleccionan columnas explicitas y excluyen password: el hash nunca viaja al cliente. insertar() aplica password_hash() con BCrypt. actualizar() solo cambia la contrasena si el campo llega no vacio.

<?php // models/Usuario.php class Usuario { private PDO $db; private string $table = 'usuario'; // password excluido deliberadamente — nunca viaja al cliente private string $campos = 'id, nombre, apellido, email, telefono, activo, created_at'; public function __construct(PDO $db) { $this->db = $db; } public function listar(): array { $stmt = $this->db->query( "SELECT {$this->campos} FROM {$this->table} ORDER BY id DESC" ); return $stmt->fetchAll(); } public function obtener(int $id): array|false { $stmt = $this->db->prepare( "SELECT {$this->campos} FROM {$this->table} WHERE id = ?" ); $stmt->execute([$id]); return $stmt->fetch(); } public function insertar(array $data): int { $hash = password_hash($data['password'], PASSWORD_BCRYPT); $stmt = $this->db->prepare( "INSERT INTO {$this->table} (nombre, apellido, email, password, telefono) VALUES (:nombre, :apellido, :email, :password, :telefono)" ); $stmt->execute([ ':nombre' => $data['nombre'], ':apellido' => $data['apellido'], ':email' => $data['email'], ':password' => $hash, ':telefono' => $data['telefono'] ?? null, ]); return (int) $this->db->lastInsertId(); } public function actualizar(int $id, array $data): int { // Si viene password se hashea y actualiza; si no, se conserva el existente if (!empty($data['password'])) { $stmt = $this->db->prepare( "UPDATE {$this->table} SET nombre=:nombre, apellido=:apellido, email=:email, password=:password, telefono=:telefono WHERE id = :id" ); $params = [ ':nombre' => $data['nombre'], ':apellido' => $data['apellido'], ':email' => $data['email'], ':password' => password_hash($data['password'], PASSWORD_BCRYPT), ':telefono' => $data['telefono'] ?? null, ':id' => $id, ]; } else { $stmt = $this->db->prepare( "UPDATE {$this->table} SET nombre=:nombre, apellido=:apellido, email=:email, telefono=:telefono WHERE id = :id" ); $params = [ ':nombre' => $data['nombre'], ':apellido' => $data['apellido'], ':email' => $data['email'], ':telefono' => $data['telefono'] ?? null, ':id' => $id, ]; } $stmt->execute($params); return $stmt->rowCount(); } public function eliminar(int $id): int { $stmt = $this->db->prepare("DELETE FROM {$this->table} WHERE id = ?"); $stmt->execute([$id]); return $stmt->rowCount(); } }

Paso 3.6: index.php — router principal

Maneja CORS, extrae el recurso e id de la URI, y delega a la funcion correspondiente. El segmento api_ucv del path de Laragon se elimina automaticamente antes de leer el recurso.

<?php // index.php — router principal header("Content-Type: application/json; charset=UTF-8"); header("Access-Control-Allow-Origin: *"); header("Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS"); header("Access-Control-Allow-Headers: Content-Type, Authorization"); // Responder preflight CORS sin procesar nada mas if ($_SERVER['REQUEST_METHOD'] === 'OPTIONS') { http_response_code(204); exit; } require_once 'config/Database.php'; require_once 'models/Usuario.php'; $db = (new Database())->getConnection(); $usuario = new Usuario($db); $method = $_SERVER['REQUEST_METHOD']; // Extraer segmentos de la URI limpiando el prefijo del proyecto // Laragon: /api_ucv/usuarios/5 -> ['usuarios', '5'] $uri = trim(parse_url($_SERVER['REQUEST_URI'], PHP_URL_PATH), '/'); $segments = explode('/', $uri); // Eliminar el segmento del nombre del proyecto si Laragon lo incluye en la ruta $projectFolder = 'api_ucv'; if (isset($segments[0]) && $segments[0] === $projectFolder) { array_shift($segments); } $resource = $segments[0] ?? ''; $id = isset($segments[1]) && $segments[1] !== '' ? (int) $segments[1] : null; $body = json_decode(file_get_contents("php://input"), true) ?? []; // ----- Router ----- switch ($resource) { case 'usuarios': routeUsuarios($usuario, $method, $id, $body); break; default: http_response_code(404); echo json_encode(['error' => 'Recurso no encontrado']); } // ----- Handlers ----- function routeUsuarios(Usuario $u, string $method, ?int $id, array $body): void { switch ($method) { case 'GET': if ($id) { $data = $u->obtener($id); echo $data ? json_encode($data) : (http_response_code(404) && json_encode(['error' => 'Usuario no encontrado'])); } else { echo json_encode($u->listar()); } break; case 'POST': if (empty($body['nombre']) || empty($body['apellido']) || empty($body['email']) || empty($body['password'])) { http_response_code(422); echo json_encode(['error' => 'nombre, apellido, email y password son obligatorios']); return; } $newId = $u->insertar($body); http_response_code(201); echo json_encode(['id' => $newId, 'mensaje' => 'Usuario creado']); break; case 'PUT': if (!$id) { http_response_code(400); echo json_encode(['error' => 'Se requiere el id en la URL']); return; } if (empty($body['nombre']) || empty($body['apellido']) || empty($body['email'])) { http_response_code(422); echo json_encode(['error' => 'nombre, apellido y email son obligatorios']); return; } $rows = $u->actualizar($id, $body); echo json_encode(['filas' => $rows, 'mensaje' => 'Usuario actualizado']); break; case 'DELETE': if (!$id) { http_response_code(400); echo json_encode(['error' => 'Se requiere el id en la URL']); return; } $rows = $u->eliminar($id); echo json_encode(['filas' => $rows, 'mensaje' => 'Usuario eliminado']); break; default: http_response_code(405); echo json_encode(['error' => 'Metodo no permitido']); } }

Dos cosas importantes en este archivo

PARTE 5: Modelo, API Service y Retrofit Client

Paso 5.1: model/Usuario.kt

Crea el archivo en java/com.example.apiucv/model/Usuario.kt. Los nombres de los campos deben coincidir exactamente con las claves JSON que devuelve la API. @SerializedName se usa cuando el nombre en JSON difiere del nombre en Kotlin (aqui created_at vs createdAt). Paso 5.3: network/ApiService.kt

Crea el archivo en java/com.example.apiucv/network/ApiService.kt. Cada funcion corresponde a un endpoint de la API. La anotacion indica el metodo HTTP y la ruta relativa a BASE_URL. Todas son suspend para llamarlas desde coroutines.

package com.example.apiucv.network import com.example.apiucv.model.Usuario import retrofit2.Response import retrofit2.http.* interface ApiService { @GET("usuarios") suspend fun listarUsuarios(): Response<List<Usuario>> @GET("usuarios/{id}") suspend fun obtenerUsuario(@Path("id") id: Int): Response<Usuario> @POST("usuarios") suspend fun crearUsuario(@Body usuario: Usuario): Response<Map<String, Any>> @PUT("usuarios/{id}") suspend fun actualizarUsuario( @Path("id") id: Int, @Body usuario: Usuario ): Response<Map<String, Any>> @DELETE("usuarios/{id}") suspend fun eliminarUsuario(@Path("id") id: Int): Response<Map<String, Any>> }

Paso 5.4: network/RetrofitClient.kt

Crea el archivo en java/com.example.apiucv/network/RetrofitClient.kt. Es un object (singleton) que crea la instancia de Retrofit una sola vez. El HttpLoggingInterceptor imprime en Logcat cada peticion y respuesta HTTP completa, lo cual es muy util para depurar durante el desarrollo.

package com.example.apiucv.network import okhttp3.OkHttpClient import okhttp3.logging.HttpLoggingInterceptor import retrofit2.Retrofit import retrofit2.converter.gson.GsonConverterFactory object RetrofitClient { private val loggingInterceptor = HttpLoggingInterceptor().apply { level = HttpLoggingInterceptor.Level.BODY } private val okHttpClient = OkHttpClient.Builder() .addInterceptor(loggingInterceptor) .build() val apiService: ApiService by lazy { Retrofit.Builder() .baseUrl(AppConfig.BASE_URL) .client(okHttpClient) .addConverterFactory(GsonConverterFactory.create()) .build() .create(ApiService::class.java) } }

PARTE 8: Cambiar de entorno local a VPS

Lo que cambia al pasar a produccion

La arquitectura no cambia en absoluto. Solo cambian dos cosas: la URL base en Android y el archivo .env en el servidor.

En el servidor (VPS)

  1. Sube la carpeta api/ al servidor (excepto .env).
  2. Crea el archivo .env directamente en el servidor con las credenciales reales.
  3. Verifica que el dominio apunte al directorio correcto.
  4. Activa HTTPS con Let's Encrypt o el proveedor del hosting.

En Android

  1. Edita AppConfig.kt y cambia BASE_URL a la URL del VPS.
  2. Quita android:usesCleartextTraffic="true" del Manifest si la URL es HTTPS.
  3. Sincroniza y genera el APK.

En .env del VPS

APP_ENV=production APP_DEBUG=false DB_HOST=127.0.0.1 DB_NAME=app_usuarios DB_USER=usuario_bd DB_PASS=contrasena_fuerte

Buenas practicas de seguridad en produccion

  • Usa HTTPS en todos los endpoints.
  • No uses el usuario root de MySQL en produccion. Crea un usuario con permisos minimos.
  • Valida y sanea todos los datos recibidos en la API antes de usarlos en SQL.
  • Usa siempre consultas preparadas con PDO (ya incluidas en el modelo de esta guia).
  • El archivo .env nunca debe estar en el repositorio ni ser accesible desde el navegador.
  • Desactiva APP_DEBUG en produccion para no exponer trazas de error.

Checklist final

  • La API responde correctamente en Postman antes de integrarla a Android.
  • .env no esta en el repositorio y esta bloqueado en .htaccess.
  • Todas las consultas SQL usan sentencias preparadas con PDO.
  • La contrasena se hashea con password_hash() antes de insertarla en MySQL.
  • El campo password nunca aparece en las respuestas GET de la API.
  • En el dialogo de edicion, dejar el campo vacio conserva la contrasena existente.
  • Android agrega el permiso de Internet en el Manifest.
  • La URL base esta centralizada en AppConfig.kt.
  • Las llamadas de red se hacen con lifecycleScope.launch para no bloquear el hilo principal.
  • En produccion se usa HTTPS y se elimina usesCleartextTraffic.
